# 企业级 AI 问答系统 — 面试题与参考答案

> 基于项目源码生成，覆盖系统架构、MCP 协议、RAG、Agent/ReAct、记忆系统、后端、前端、数据库等方向。

---

## 一、系统架构

**1. 请介绍一下这个项目的整体架构，各层的职责是什么？**

本系统采用五层架构：

- 前端展示层：Vue 3 + TypeScript + Element Plus，聊天界面、文档管理，axios 发 REST，原生 fetch 消费 SSE。
- API 网关层：FastAPI + Uvicorn（port 8001），路由分发、CORS、X-Trace-ID 追踪、全局异常处理。
- Agent 核心层：QAAgent（ReAct 多轮工具调用）、DocumentAgent、MemoryAgent、OrchestratorAgent，Tool 系统，Skill 引擎。
- MCP 协议层（v1.1.0 新增）：MCPDispatcher + KnowledgeMCPServer + DocumentMCPServer，将系统能力以 JSON-RPC 2.0 标准对外暴露。
- 基础设施层：ChromaDB（向量存储）、MongoDB（会话/文档/关键词索引）、EmbeddingCache（LRU）。

---

**2. 系统使用了 SSE 实现流式响应，为什么选择 SSE 而不是 WebSocket？**

问答是典型的单向流，服务端持续推送文本片段，客户端只需接收，不需要双向通信。SSE 基于 HTTP，实现简单，浏览器自动重连，兼容标准 HTTP 负载均衡（WebSocket 需要粘性会话）。

项目中前端用 `fetch` + `ReadableStream` + `TextDecoder` 手动解析 SSE 而非 EventSource，原因是需要支持 `AbortController` 中止流（用户点击停止按钮），EventSource 不支持主动中止。

---

**3. FastAPI 启动时做了哪些初始化工作？顺序是否重要？**

顺序：ChromaDB 初始化 → MongoDB 连接 → Tool 导入（触发 ToolRegistry 注册）→ Agent 注册 → 后台文件扫描器 → MCP Server 模块导入（模块级自动注册）。

顺序非常重要：Tool 必须在 Agent 之前注册，因为 QAAgent 初始化时会调用 `ToolRegistry.get_tools_for_llm()`；ChromaDB 和 MongoDB 必须在 Agent 之前就绪，因为 Agent 处理请求时依赖这两个存储。

---

**4. 系统如何实现请求链路追踪？**

中间件为每个请求生成或透传 `X-Trace-ID`（UUID），存入 `request.state.trace_id`，写入响应头，异常处理器将其写入错误响应体。定位问题时从前端错误提示或响应头取 trace_id，在日志中搜索完整链路。

## 二、MCP 协议（v1.1.0 新增）

**5. 什么是 MCP 协议？项目中是如何实现的？**

MCP（Model Context Protocol）是一种标准化协议，允许 LLM 应用以统一方式暴露工具（Tools）、资源（Resources）和提示词模板（Prompts），任意 MCP 兼容客户端都可以发现并调用这些能力。

项目实现分四层：
- `protocol.py`：JSON-RPC 2.0 消息模型（JSONRPCRequest/Response/Error）+ MCP 业务模型（InitializeResult、CallToolResult、GetPromptResult 等），协议版本 `2024-11-05`。
- `base.py`：MCPServer 基类（注册/列出 tools/resources/prompts，抽象 execute_tool/read_resource/get_prompt）+ MCPServerRegistry（类变量字典，模块导入时自动注册）。
- `dispatcher.py`：MCPDispatcher 路由 8 个 JSON-RPC 方法，聚合所有注册 Server 的能力，notification（id=None）处理后返回 None（调用方返回 204）。
- `transport.py` + `router.py`：HTTP POST 端点解析请求体 → 校验 → dispatch；SSE 端点建立持久连接，15s keepalive；`/mcp/capabilities` 调试端点。

---

**6. MCPDispatcher 如何处理 tools/call 请求？工具不存在时返回什么？**

遍历 `MCPServerRegistry` 中所有 Server，找到包含目标工具名的 Server，调用 `server.execute_tool(name, arguments)`。如果没有任何 Server 包含该工具，抛出 `ToolNotFoundError`，dispatcher 捕获后返回错误码 `-32003`（TOOL_NOT_FOUND）。工具执行异常时返回 `CallToolResult(isError=True)`，不抛出 JSON-RPC 错误。

---

**7. MCP 的 SSE 端点有什么作用？为什么需要 keepalive？**

SSE 端点建立持久连接，服务端可主动推送消息给客户端（如工具执行结果、进度通知）。连接建立时推送 `endpoint` 事件告知客户端 POST 地址。

keepalive（每 15s 发送 `: keepalive\n\n` 注释行）的作用：防止代理服务器（Nginx/负载均衡器）因连接空闲超时而断开 TCP 连接，同时也用于检测客户端是否已断开（`await request.is_disconnected()`）。

---

## 三、RAG 与检索

**8. 项目的检索流程是怎样的？为什么不用混合检索（HybridRetriever）？**

当前 RAG 流程：查询向量化（含关键词增强：`query + keywords[:3]`）→ ChromaDB 向量检索 → AnswerValidator 相似度过滤（阈值 0.7，最多保留 5 个文档）→ 构建上下文 → LLM 生成。

原 HybridRetriever（向量 0.6 + BM25 0.4，RRF 融合）在代码清理中被移除，因为它依赖内存 BM25 索引，不支持持久化，且实际未被 QAAgent 调用。现在关键词检索改为基于 MongoDB 的倒排索引（`keyword_index` 集合），通过 `keyword_search()` 函数独立调用。

---

**9. AnswerValidator 的相似度过滤是如何工作的？**

ChromaDB 返回的距离（distance）转换为相似度：`similarity = 1 - distance / 2`（假设距离范围 0-2）。低于 `min_similarity_score`（默认 0.7）的文档被过滤掉，最多保留 `max_relevant_docs`（默认 5）个文档。这两个参数可通过约束配置接口动态调整。

---

**10. KBQueryOptimizer 的查询优化逻辑是什么？**

通过 LLM 分析用户查询，提取关键词，改写为更适合向量检索的形式，同时返回置信度。只有当 `confidence > 0.6` 时才使用优化后的查询，否则使用原始查询，避免低质量改写反而降低检索效果。优化结果会被缓存（QueryCache），避免相同查询重复调用 LLM。

---

## 四、Agent 与 ReAct

**11. 项目的 QAAgent 实现了 ReAct 模式，请详细描述其工作流程。**

ReAct（Reason → Act → Observe）循环，最多 5 轮：

1. Reason：调用 `LLMClient.chat_with_tools()` 判断是否需要工具，返回 `ToolDecision`。
2. Act：`ToolExecutor.execute_batch()` 并行执行本轮所有工具调用。
3. Observe：将工具调用和结果以 `tool_calls` + `tool` 消息格式追加到消息链。
4. 再次 Reason：基于所有 Observation 判断是否继续调用工具。
5. 退出条件：LLM 不再调用工具，或达到 MAX_ITERATIONS=5。
6. 最终生成：基于完整消息链（含所有工具结果）流式生成回答。

与单轮工具调用的区别：多轮允许 LLM 根据前一轮工具结果决定下一步行动，实现更复杂的推理链。

---

**12. Tool 决策为什么使用低温度（0.1）？生成回答为什么用更高温度（0.3）？**

Tool 决策需要精确判断"是否调用工具"和"调用哪个工具"，这是一个分类决策，需要稳定一致的输出，低温度（0.1）减少随机性，确保相同查询得到相同决策。

生成回答需要一定的语言多样性和流畅性，适中温度（0.3）在保证准确性的同时避免回答过于机械。

---

**13. ToolRegistry 是如何工作的？Tool 是如何注册的？**

ToolRegistry 采用类变量字典 `_tools` 存储，Tool 实现类在模块导入时通过装饰器或显式调用自动注册。`get_tools_for_llm()` 方法将所有启用的 Tool 转换为 OpenAI function calling 格式（包含 name、description、parameters JSON Schema），供 LLM 决策使用。

---

## 五、记忆系统

**14. HybridMemoryService 如何实现短期记忆和长期记忆的融合？**

`build_context()` 方法：
- 短期记忆：`ShortTermMemory`（内存，max_tokens=3000），首次访问时从 MongoDB 加载历史消息，Token 超限时自动截断旧消息。
- 长期记忆：并行执行两个检索任务——`RAGRetriever.retrieve_documents()`（知识库文档，top_k=3）和 `_retrieve_related_conversations()`（ChromaDB conversations 集合，跨会话语义检索，过滤当前 session_id）。
- 融合：长期记忆以 `[相关上下文]` system 消息注入，短期记忆追加在后，共同构成 LLM 的输入上下文。

---

**15. 对话历史是如何向量化存储的？有什么用途？**

每次 assistant 回答后，将 `"Q: {question}\nA: {answer}"` 格式的 QA 对向量化，存入 ChromaDB conversations 集合，metadata 包含 session_id 和 type="qa_pair"。

用途：跨会话语义检索——当用户提出新问题时，可以检索历史对话中语义相关的 QA 对，作为长期记忆注入上下文，帮助 LLM 理解用户的历史偏好和已讨论过的话题。

---

## 六、后端工程

**16. 项目中的来源（sources）是如何限制数量的？**

`ResponseBuilder.MAX_SOURCES = 5`，在两处生效：
- `build_sources_from_documents()`：遍历时切片 `documents[:MAX_SOURCES]`，限制 Tool 流程中单次工具结果的来源数。
- `_extract_sources_from_tool_results()`：聚合所有工具结果后切片 `sources[:MAX_SOURCES]`，限制多轮工具调用的总来源数。
- RAG 流程：本身只取最相关的一个文档作为来源，不受此限制影响。

---

**17. 项目中有哪些缓存机制？分别解决什么问题？**

- EmbeddingCache（LRU，maxsize=1000）：缓存文本向量化结果，避免相同文本重复调用 DashScope API，降低延迟和费用。
- QueryCache：缓存 KBQueryOptimizer 的查询优化结果，避免相同查询重复调用 LLM 进行改写。
- ShortTermMemory（内存）：缓存会话的对话历史，避免每次请求都从 MongoDB 加载。

---

**18. 为什么删除了 security.py、telemetry.py、performance_monitor.py 等文件？**

这些文件实现了完整的功能（JWT/RBAC 权限系统、OpenTelemetry 链路追踪、性能监控装饰器），但没有任何路由或业务代码实际调用它们——没有路由使用 `Depends(get_current_user)`，没有代码调用 `@trace_llm_call`，没有代码调用 `@monitor_performance`。保留这些文件会增加维护负担和误导性，属于典型的死代码，清理后代码库更清晰。

---

## 七、前端工程

**19. 前端如何处理 SSE 流式响应？为什么不用 EventSource？**

使用原生 `fetch` + `ReadableStream` + `TextDecoder` 手动解析 SSE 数据流，逐行解析 `data: {...}` 格式，根据 `type` 字段分发处理（text 追加内容、done 更新来源和快捷提问、error 展示错误）。

不用 EventSource 的原因：EventSource 不支持 `AbortController`，无法在用户点击"停止"按钮时主动中止请求；另外 EventSource 不支持自定义请求头（如 Authorization）。

---

**20. Pinia store 中的 currentRelatedLinks 是如何实现跨消息去重合并的？**

`currentRelatedLinks` 是会话级别的相关链接状态，每次收到新消息的 `done` 事件时，将新链接与现有链接合并，通过链接 `id` 或 `url` 去重，确保同一会话内多次问答产生的相关链接不重复展示。

---

## 八、数据库

**21. 为什么同时使用 ChromaDB 和 MongoDB？各自负责什么？**

两者职责互补：
- ChromaDB：专门的向量数据库，负责语义检索（HNSW 索引，cosine 相似度），存储文档 chunk 向量和对话历史向量，支持 `query_embeddings` 相似度查询。
- MongoDB：通用文档数据库，负责结构化数据持久化——会话、消息、文档状态、外部链接、关键词倒排索引。Motor 提供异步驱动，支持连接池。

如果只用 MongoDB，向量检索需要全量扫描，性能差；如果只用 ChromaDB，无法高效存储和查询结构化的会话/消息数据。

---

**22. MongoDB 降级策略是什么？**

`connect_to_mongo()` 失败时，系统不会崩溃，而是降级为纯短期记忆模式：`HybridMemoryService` 仍然可以使用内存中的 `ShortTermMemory`，只是无法持久化消息到数据库，也无法加载历史消息。这保证了系统在数据库不可用时仍能提供基本的问答服务。
